# Fastlane 채점 보정 테스트 — 시니어 서비스 디자인

3-Pass (Draft → Skeptic → Judge) 파이프라인 + rubric 자동 생성을 **UX/서비스 디자인 도메인** 에서 검증. 알고리즘 / 공익광고와 다른 어휘 (페르소나·페인포인트·유저 저니·MVP·접근성) 가 rubric 에 제대로 나타나는지 확인 목적.

도메인: **서비스 디자인 - 1인 가구 시니어 일상 지원**

### 평가 항목 (가중치 합 100%)

| 항목명 | 가중치 | 채점 기준 설명 |
|---|---|---|
| 사용자 통찰의 깊이 | 30 | 시니어 1인 가구의 실제 페인포인트와 일상 맥락에 대한 구체적 이해 |
| 솔루션 적합성 | 30 | 페르소나의 페인포인트와 제안 솔루션 사이의 인과 일치도 |

## 2️⃣ 출품작 A — `~10점 예상` (정보 부족)

### 제목
```
노인용 앱
```

### 팀명 / 출품자
```
학생B
```

### 본문 / 요약
```
독거 노인들이 외로워서 잘 안먹는다. 영양 관리 앱 만들거. 음식 알려주고 친구 만나게도 해줄거. 안드로이드로 만듦.
```

### 예상 채점 흐름
- **완전성 미충족**: 페르소나 X / 페인포인트 검증 데이터 X / 솔루션 인과 X / MVP 범위 X / 접근성 고려 X / KPI X
- **Draft** → ~20 점 (한 줄 컨셉, anchor ~10/~25 와 매칭)
- **Skeptic** → 과장 지적: "노인용 앱" 이라는 표현 자체가 페르소나·접근성 검증 부재. "독거 노인 외로움 → 영양 관리 앱" 인과가 단순 가정 → suggested ~12
- **Judge** → **15-22점대**

## 3️⃣ 출품작 B — `~90점 예상` (모든 요건 충족)

### 제목
```
오늘 한 끼 — 시니어 1인 가구를 위한 동네 식탁 예약 + 동행 매칭
```

### 팀명 / 출품자
```
동네한끼 스튜디오
```

### 본문 / 요약
```
[페르소나 — 김영자 (가명, 72세, 서울 동대문구 거주)]
배우자와 사별 후 7년째 1인 가구. 자녀는 분당·미국 거주. 무릎 통증으로 30분 이상 외출 어려움. 끼니는 하루 1-2회, 그마저 김치찌개 같은 단일 메뉴 반복 (반찬 1-2가지). 동주민센터의 경로식당은 인지하지만 "혼자 가기 민망해서" 한 달에 1-2회만 이용. 스마트폰은 카카오톡·전화·뉴스 정도 사용, 음식 사진은 못 찍음.

[페인포인트 — 3중 결합]
1. 신체적 — 시장·식당 왕복이 부담스러움 (반경 300m 이내만 활동)
2. 정서적 — "혼자 먹는 식사" 가 무의미하게 느껴져 끼니 자체를 거름. 보건복지부 2024 조사에서 65세+ 1인 가구 우울 위험군 31.2%, 그중 "식사 의욕 없음" 응답 62%.
3. 사회적 — 같은 동네에 비슷한 처지의 이웃이 있어도 "내가 먼저 말 걸기는…" 의 자기검열로 접점 형성 안 됨.

[기존 서비스 한계]
- 경로식당: 메뉴 제공만, 동행 매칭 없음. 혼자 가기 부담.
- 시니어 도시락 배달: 식사 자체는 해결되지만 사회적 고립 강화.
- 노인복지센터 프로그램: 일정 고정·이동 부담·온보딩 진입 장벽 큼.

[솔루션 — "오늘 한 끼"]
시니어가 동네 단골 식당(점심 11:30-13:00) 에 한 번 클릭으로 식사 예약, 같은 시간 같은 식당에 다른 시니어가 예약하면 자동 매칭 — "오늘은 김 어르신과 같이 드시겠어요?" 알림. 거절 자유, 부담 없음. 식당은 시니어 단가로 사전 계약 (지자체 보조금 매칭).

[MVP 범위 — 8주 시범]
- 지역: 서울 동대문구 1개 동 (시니어 1인 가구 420세대)
- 식당: 동주민센터와 협약한 3-5개 일반식당 (메뉴 단일가 8천원, 지자체 4천원 보조)
- 기능: 예약 (큰 버튼 3개 UI) / 자동 동행 매칭 / 위치 안내 (도보 5분 이내만)
- 미포함: 결제 (현장 결제 + 보조금 정산은 추후 통합), 다국어 (필요 시 v2)

[UI 디자인 원칙 — 시니어 접근성]
- 한 화면 1 action — "오늘 점심 예약" / "내일로 미루기" 두 버튼만
- 글자 18pt+, 대비 7:1 이상
- 음성 안내 옵션 (TTS)
- 카카오톡 채널 기반 — 별도 앱 설치 X (디지털 진입 장벽 회피)

[기술·운영 모델]
- 카카오톡 챗봇 (KakaoWork Open API) + 백엔드 Supabase
- 매칭 알고리즘: 단순 그리디 (같은 시간·식당·도보거리 5분 이내). 복잡 ML 불필요.
- 동주민센터 사회복지사 1명이 운영 모니터링 (이상 신호: 며칠 연속 결식, 매칭 거절 반복)

[KPI 및 측정 — 8주 후]
- 1차 KPI: 참여 시니어의 주간 식사 빈도 변화 (기준 7.2회 → 목표 12회)
- 2차 KPI: 동행 매칭 성공률 (목표 40%+, 시도 대비)
- 3차 KPI: 8주 시점 GDS-15 (노인우울척도) 변화 — 기준 대비 평균 −2점 이상
- 부수 측정: 매칭 시니어의 자발적 반복 매칭 비율 (관계 형성 지표)

[차별점 — 왜 새로운가]
경로식당(메뉴만), 도시락 배달(고립 강화), 복지센터(진입 장벽) 모두 "식사" 또는 "사회적 접촉" 둘 중 하나만 해결한다. 본 서비스는 두 페인포인트를 같은 트리거(점심 예약) 안에 묶어 — 동행 매칭이 부담스러운 별도 행동이 아니라 식사 예약의 부수효과로 발생하도록 설계했다. 우울 위험군의 자기검열 메커니즘 ("내가 먼저는…") 을 자동 매칭으로 우회한다.

[확장 — 12개월 로드맵]
1-3개월: 1개 동 시범 / 4-6개월: 동대문구 3개 동 확장 + 결제 통합 / 7-12개월: 서울 5개 자치구 확대 + 영양사 추천 식단 컨설팅 옵션. 12개월 누적 참여 시니어 3,000명, GDS-15 평균 개선 −2.5점 목표.
```

### 예상 채점 흐름
- **완전성 충족**: 페르소나 ✅ (구체 인물 + 데이터) / 페인포인트 ✅ (3중 결합 + 보건복지부 인용) / 솔루션 인과 ✅ / MVP ✅ (8주, 1개 동, 기능 명세) / 접근성 ✅ (시니어 UI 원칙) / KPI ✅ (3단계 + GDS-15) / 차별점 ✅
- **Draft** → 82~90 (사용자 통찰: 자기검열 메커니즘 / 적합성: 식사 트리거로 사회 접촉 자동화 / 실현성: 카카오톡 기반 + 단순 그리디 / 임팩트: GDS-15 측정 가능)
- **Skeptic** → 명시적 근거 다수. 작은 보정 ("동행 매칭 거절률 가설은 시범에서 검증 필요" 정도) → suggested ~85
- **Judge** → **86-92점대**

- [ ] A 점수 < 30 / B 점수 > 80 / 격차 > 50
- [ ] **rubric 자체가 UX 어휘** 로 작성됐는지 확인 (페르소나/페인포인트/유저 저니/접근성/MVP)
  - startup 어휘 (TAM/CAC) 또는 광고 어휘 (CPM) 가 끼면 도메인 lens 가 흐려진 것
- [ ] B 의 axis_reports 가 700-1200자 깊이로 작성
- [ ] B 의 종합 리포트 (8 섹션, 2000-3500자) 에 cross-axis 통찰 포함
  - 예상: "사용자 통찰의 깊이 (자기검열 메커니즘 발견) + 솔루션 적합성 (식사를 트리거로 사회 접촉 자동 발생) 이 결합되어 기존 서비스의 분절성 (식사 OR 사회 둘 중 하나만) 을 우회"
- [ ] B 의 심사위원 권장 검토 포인트 (3-5개) 가 적절
  - 예상: "매칭 거절률 가설의 시범 데이터 검증 / 사회복지사 1인당 모니터링 가능 시니어 수 / GDS-15 측정의 자기보고 한계"
- [ ] B 의 출품자 보완 방향이 실행 가능
- [ ] 각 axis 의 "심층 분석 보기" 클릭 시 펼침 동작
- [ ] A 와 B 모두 같은 rubric (같은 대회) 로 평가됨 — 잣대 일관성 검증

## 5️⃣ 도메인 어휘 검증 (rubric 생성 직후 확인)

대회 생성 후 detail 페이지의 "평가표" 섹션에서 각 axis 의 rubric 을 펼쳐서 확인:

### 사용자 통찰의 깊이 (예상 도메인 어휘)
- 페르소나 / 페인포인트 / 유저 저니 / 행동 관찰
- 컨텍스트 인터뷰 / 에쓰노그라피
- 일상 맥락 / 자기검열 / 진입 장벽

### 솔루션 적합성
- 페인포인트 → 솔루션 인과 / Job-to-be-done
- 단일 vs 결합 솔루션 / 우회 vs 직접 해결
- 사용자 트리거 설계

### 실현 가능성
- MVP 범위 정의 / scope creep
- 접근성 (a11y) — 시니어 디지털 진입 장벽
- 운영 모델 (사회복지사 / 지자체 협업)
- 매칭 알고리즘 단순성 (over-engineering 회피)

### 사회 임팩트
- 측정 가능 outcome (GDS-15, 식사 빈도 등)
- 도달 인구 vs 기존 서비스의 분절성
- 자기보고 vs 객관 지표

위 어휘들이 rubric 본문에 자연스럽게 등장해야 함. 만약 "ACV", "유닛 이코노믹스" 같은 startup 어휘가 끼면 rubric_generator 의 도메인 lens 가 흐려진 것.

## 6️⃣ 보정 시나리오

### A 가 50+ 나오면
→ rubric 의 완전성 체크포인트가 효력을 못 내는 거. "재생성" 버튼으로 rubric 새로 만들기.

### B 가 60대 나오면
→ 너무 짜졌음. `prompts_competition/rubric_generator.md` 의 ~88 anchor 의 "상위 5%" 표현을 "상위 10-15%" 로 완화.

### 도메인 어휘가 startup 류로 새 나오면
→ `prompts_competition/rubric_generator.md` 의 "theme 으로 들어온 도메인을 그대로 받아라. 일반 startup 평가 같은 fallback 으로 도망치지 마라" 룰이 안 먹은 것. 더 명시적으로 강조하거나, 도메인별 예시 추가.

## 7️⃣ 비고

- 이 도메인은 startup/광고와 달리 **수익 모델·도달 지표가 핵심이 아님**. rubric_generator 가 그 차이를 인지하고 임팩트/접근성 어휘로 갈아끼우는지 검증 핵심.
- 같은 출품작을 여러 번 평가해도 점수 드리프트가 ±5 이내 (3-Pass 결정성).
- 다른 도메인 테스트 케이스는 같은 폴더의 다른 `fastlane_*` 파일 참고:
  - [fastlane_scoring_calibration_tests.md](./fastlane_scoring_calibration_tests.md) — 공익광고 (청년 정신건강)
